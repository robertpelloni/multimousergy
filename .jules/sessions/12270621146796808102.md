# MultiMousergy (formerly NetMux) - Session Memory & Architectural Observations

## Project Overview
MultiMousergy is a decoupled, network-based multi-cursor system designed to seamlessly bridge independent physical PCs. It allows local mouse hardware to cross network/display boundaries while maintaining a cohesive 3D-composited spatial workspace (currently shifting towards WebRTC/DXGI capabilities). The core conceit involves suppressing native input at screen edges and injecting network-received input into remote systems via kernel-level virtual drivers, tricking the OS into treating the network cursor as a genuine secondary hardware mouse.

## Core Architecture & Design Patterns
1. **Two-Layer Decoupling Strategy:**
   * **Capture & Send:** Uses low-level OS hooks (Windows: `WH_MOUSE_LL` / `WM_INPUT`, Linux: `XQueryPointer` / `evdev`) to intercept physical mouse and keyboard inputs and suppress local boundary movement.
   * **Receive & Inject:** Bypasses OS native cursor limits by writing coordinates directly to a Virtual HID Driver (ViGEmBus/Interception on Windows, `uinput`/`evdev` on Linux). 

2. **Network & Synchronization (Hybrid Protocol):**
   * **UDP/TCP Split:** The framework aggressively optimizes bandwidth using lightweight, header-only UDP packets for high-frequency absolute cursor movement, while relying on TCP for state-critical actions (clicks, authentication, clipboard sync, file transfers).
   * **Authoritative MasterStateSync:** To counter UDP packet loss and drift, the system relies on a Server-side authoritative state loop. The `SyncModule` manages peer lists, normalizes coordinates using virtual screen metrics and dynamic DPI scales, and maintains a unified jitter buffer/timeline.
   * **Timestamp-First Conflict Resolution:** Ownership of interactions (e.g., simultaneous clicks on the same UI element) is mediated via a distributed clock synchronization.

3. **Advanced Rendering & Compositing Engine:**
   * Transitions from a basic transparent window to a **Spatial 3D Composited Viewport** (`SpatialViewport`).
   * Uses Direct3D 11 (`D3D11Overlay`) for hardware-accelerated sprite rendering, texture mapping for custom cursors, and matrix math (`XMMATRIX`) to handle smooth pan/zoom camera transitions between the local and remote desktop representations.
   * Cross-platform considerations are maintained strictly with `#ifdef _WIN32` and `#ifdef __linux__` (X11 rendering/clipboard).

4. **Security & Data Integrity:**
   * **Mutual Authentication:** `AuthModule` uses SHA-256 for challenge-response authentication. It utilizes constant-time string comparisons to prevent timing attacks.
   * **Replay Protection:** Incorporates sequence numbering to discard stale or malicious duplicate packets.
   * **High-Capacity Chunking:** Uses a multi-part packet protocol for transferring large payloads (Unicode Clipboard, File Drag-and-Drop) with SHA-256 streaming verification.

5. **Structural Shifts & Recent Implementations (v0.1.71 -> v0.1.72-alpha):**
   * **WebRTC & Spatial Media Integration:** The system is aggressively moving toward low-latency integrated video chat and spatial desktop projection. We recently implemented architectural stubs:
     * `WebRTCManager`: Skeletons for SDP Offer/Answer handshakes, DataChannels (replacing raw UDP for cursor tracking?), and Media Tracks.
     * `DesktopCapture`: DXGI Output Duplication logic initialized for capturing D3D11 frames.
     * `WebcamCapture`, `VideoEncoder`, `VideoDecoder`: Scaffolding utilizing Windows Media Foundation (MFT) for hardware H.264 video processing.
   * **Connection Resilience Refactor:** `NetworkManager` is transitioning from blocking connection patterns to a non-blocking, state-driven machine (using `select()`), greatly reducing CPU spin-loop overhead and UI freezing.

## UI & Personalization Rules
* **ConfigGUI** handles the system tray, tabbed settings, minimap layout, and machine telemetry (IP/Hostname). It uses Win32 native controls. 
* All new framework capabilities (colors, themes, connection status, file transfers) are tightly bound to the UI layer so users have 1:1 visibility of Replay attacks, Peer state, and drift telemetry. 
* Care must be taken to include `winsock2.h` properly within `#ifdef _WIN32` blocks in UI modules to avoid breaking Linux build compatibilities via CMake.

## Next Steps for Successor Models
* **Flesh out WebRTC Skeletons:** The project has the headers and empty implementations for WebRTC. The immediate goal should be linking a native `libwebrtc` instance to generate actual SDPs and ICE candidates.
* **DXGI Frame Forwarding:** `DesktopCapture` successfully hooks IDXGIOutputDuplication. The acquired `ID3D11Texture2D` frames need to be fed into the `VideoEncoder` or drawn onto the `SpatialViewport` planes.
* Maintain rigorous `#ifdef` guardrails, as the project strives to compile on both Windows (Visual Studio) and Linux (GCC via CMake). Always verify X11/Linux build chains when modifying generic framework components.

# MultiMousergy (formerly NetMux) - Project Architecture & Memory

## 1. Core Vision & Concept
MultiMousergy is a highly advanced, decentralized multi-cursor system that allows independent PCs to share mouse and keyboard input across network boundaries. Unlike standard KVM tools, it does not hijack the remote user's cursor. Instead, it virtualizes hardware input to create a cohesive, shared spatial workspace where remote cursors exist independently alongside local ones.

## 2. Decoupled Two-Layer Input Architecture
*   **Capture Layer:** Intercepts physical input and suppresses boundary transitions locally. Uses low-level OS hooks (`WH_MOUSE_LL`, `WM_INPUT` on Windows; `XQueryPointer`, `evdev` on Linux).
*   **Injection Layer:** Receives network data and bypasses OS cursor limits by writing coordinates directly to a Virtual HID Driver.
    *   *Windows:* Employs dynamic SDK loading for ViGEmBus and Interception.
    *   *Linux:* Injects via `uinput` and `evdev`.

## 3. Hybrid Networking Protocol
*   **UDP for Movement:** Utilizes lightweight, header-only UDP packets with delta compression for high-frequency cursor movement to ensure low bandwidth and ultra-low latency.
